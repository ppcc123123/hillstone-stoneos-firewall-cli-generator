HA
HA介绍
当一台设备不可用或者不能处理来自客户端的请求时，该请求会及时转到另外的可用设备来处理，这样就保
证了网络通信的不间断进行，极大地提高了通信的可靠性。实现HA功能，用户需要配置两台采用完全相同型
号的硬件平台及相同的VR特性组成HA簇。
设备支持HA的工作模式为：Active-Passive（A/P）模式和Peer Active-Active（A/A）模式：
l Active-Passive（A/P）模式：在HA簇中配置两台设备组成一个HA组，组内只有一台主设备。主设备处于活动
状态，转发报文，同时将其所有网络和配置信息以及当前会话信息传递给备份设备。当主设备出现故障时，备份
设备接替主设备工作，转发报文。这种A/P模式具有较强冗余性，而且其网络结构简单，便于维护管理。
l Peer Active-Active（A/A）模式：该模式是一种HA Active-Active模式。处于Peer A/A 模式下的两台设备都处
于主动状态且同时运行各自的工作、相互监测对方的情况。当其中一台设备发生故障时，另外一台设备运行其自
身的工作并且接管故障设备的工作。在Peer A/A 模式下，只有处于主动状态的设备的接口可以正常收发报文；
处于禁用状态的设备可使2台设备的配置信息保持一致，但其接口不收发任何报文。Peer A/A 模式配置更加灵

活，比较适合在非对称路由环境中部署。
HA基础概念
HA簇
HA簇是实现HA功能的设备的组合。对于外部网络设备而言，一个HA簇是一个单一的设备，处理网络流量和
提供安全服务。HA簇通过簇ID进行标识。为设备指定HA簇ID后，设备进入HA状态，执行HA功能。
HA组
系统会对HA簇中相同HA组ID的设备，按照HCMP协议，根据设备的HA配置，进行主备选举。主设备处于活
动状态处理网络流量，而当主设备出现故障时，其它设备代替主设备继续工作。当为安全设备设置簇ID时，
组ID为0的HA组会自动创建。在Active-Passive（A/P）模式中，设备仅具有HA组0。
HA Node
为区分HA簇中的HA设备，用户可使用HA Node值来标识设备。目前的版本仅支持Node值为0和1。
在HA Peer模式下，系统可通过标识HA Node值来决定哪个设备处于主动状态，哪个处于禁用状态。在HA
组0中，HA Node值为0的设备处于主动状态，Node值为1的设备处于禁用状态。在HA组1中，Node值为
1的设备处于主动状态，Node值为0的设备处于禁用状态。
HA组接口和虚拟MAC
在HA环境中，每个HA组都具有接口，流量通过接口进行传输。每个HA组的主设备维护对应接口的虚拟MAC
（VMAC）地址，流量通过这些具有VMAC地址的接口进行转发。HA簇中不同HA组之间不互相转发数据。
VMAC地址由HA虚拟基MAC、簇ID、HA组ID以及物理接口索引确定。
HA选举
HA簇中，拥有同样HA组ID的具有高优先级（数值越小，优先级越高）的设备会被选举为HA组的主设备。

HA同步
为保证备份设备能够在主设备失效时代替主设备工作，主设备需要与备用设备进行同步。系统使用两种方法
进行同步，分别是实时同步（热同步）和批量同步（冷同步）。当主设备刚刚选举成功时，系统会使用批量
同步方法，将主设备信息全部同步到备份设备；当配置发生变化时，系统将使用实时同步的方法将变化的信
息同步到备份设备，或者也可以手动执行命令进行批量同步。除HA相关配置和本地配置（例如，主机名称配
置），其它的配置都会被同步。
系统在批量同步配置信息时，会自动调整备份设备所有功能模块的配置顺序，使其与主设备保持一致。同步
完成后，用户可通过WebUI或CLI命令查询主备设备配置的一致性。
同步的信息类型有三种：配置信息、文件以及RDO（Runtime Dynamic Object）。RDO的具体内容主要
包括：
l 会话信息（以下类型会话信息不会同步：到设备本身的会话、隧道会话、Deny Session、ICMP会话以及
tentative会话）
l IPSec VPN信息
l SCVPN信息
l L2TP信息
l DNS缓存映射条目
l ARP表
l PKI信息
l DHCP信息
l MAC表
l Web认证信息

注意事项
1. 组HA时，设备不会对License的类型、到期时间等进行检查，会对两台HA设备的功能特性是否相同进行比对，

HA场景中NAT功能的配置限制
在HA场景中使用NAT功能时，请注意以下配置限制：
l 在Peer Active-Active（A/A）场景下，如果HA组中的两台设备配置了相同的动态端口（dynamicport）SNAT
规则，则需要开启SNAT端口分割功能（使用split-port-pool by ha-node命令），保证HA组中的两台设备分配
的端口不会重复。
l 在Peer Active-Active（A/A）场景下，如果HA组中的两台设备配置了相同的动态IP（dynamicip）SNAT规
则，由于HA组间不会同步IP的使用情况，可能会出现转换IP被复用的问题。
l NAT规则中的源地址或者目的地址不允许包含HA连接接口（心跳接口）IP地址，避免心跳报文被NAT转换引发心
跳链路通信异常。
l 在Peer Active-Active（A/A）场景下，如果HA组中的两台设备配置了转换为出接口（eif-ip或者eif-ipv6）的
SNAT规则，由于HA组中两台设备转换的IP可能不一样，HA切换后可能会出现行为不一致的问题。

HA组配置需要在HA组配置模式下进行。进入HA组配置模式，在全局配置模式下，输入以下命令：

配置抢占模式
如果将设备配置为抢占模式，一旦设备发现自己的优先级高于主设备，就会将自己升级为主设备，而原先的
主设备将变为备份设备；如果将设备设置为非抢占模式，即使设备的优先级高于主设备，它也只能在主设备
故障时代替主设备工作。在配置抢占模式时，用户还可以设置延迟时间，使备份设备在延迟时间过后升级为
主设备。配置抢占模式，在HA组配置模式下使用以下命令：
preempt [delay-time]
l
delay-time – 指定延迟时间，单位为秒。范围是1到600秒。默认值为30秒。
在HA组配置模式下使用no preempt恢复为非抢占模式。
指定发送ARP包个数

所指定的发送ARP包个数决定，当设备发生主备切换后，系统将会立即发送5个免费ARP包，后续每秒发送1
个ARP包，直到发送的ARP包个数达到该命令所指定的个数。指定发送ARP包的个数，在HA组配置模式下使

配置SNAT端口分割
系统支持SNAT端口分割功能，即两台HA设备均配置了相同的SNAT地址池，系统会按照HA Node值在两台
设备上平均分配SNAT端口资源。若关闭该功能，两台HA设备需配置不同的SNAT地址池，每台设备将独占
所有的端口资源。该功能仅在HA Peer模式下生效。
开启SNAT端口分割功能，在全局配置模式下，使用以下命令：

在全局配置模式下，使用no split-port-pool by ha-node关闭该功能。

HA连接配置
主设备与备份设备之间的同步信息以及Hello报文等都通过HA连接进行传输。
HA连接有两种类型，分别是控制连接（Control Link）和数据连接（Data Link）。控制连接必须配置，数
据连接为可选配置（对于X系列设备和K20803设备来说，数据连接也为必配项）。
l 若只配置了控制连接，两台设备间的所有数据都通过控制连接来传输，包括心跳报文和会话信息等。
l 若同时配置了控制连接和数据连接，两台设备间的心跳报文将同时在控制连接和数据连接上传输。两台设备间配
置信息的同步将在控制连接上完成，数据报文信息的同步（如会话信息的同步）将在数据连接上完成。
配置HA连接，需要首先指定HA连接接口，然后为接口指定IP地址。
注意: 配置X系列设备和K20803设备的HA连接接口时，需要注意以下几点：
l
仅支持通过HA数据连接接口同步数据信息。
l
默认情况下，系统控制模块（SCM模块）上的所有HA口（HA0和HA1）会自动被配置为HA控
制连接接口，不需要用户进行配置。连接时，按照以下方式连线：
l
主设备的主用主控模块的HA0与备份设备的主用主控模块的HA0连接；
l
主设备的主用主控模块的HA1与备份设备的备用主控模块的HA0连接；
l
主设备的备用主控模块的HA0与备份设备的主用主控模块的HA1连接；
l
主设备的备用主控模块的HA1与备份设备的备用主控模块的HA1连接。
指定HA连接接口
用户最多可以指定两个HA控制连接接口，后配置的HA连接接口可以作为先配置的HA连接接口的备份接口工
作。当先配置的HA连接接口断开连接，后配置的HA连接接口会继续传输HA报文。
X系列设备默认使用系统控制模块上的接口作为HA控制连接接口，为避免系统控制模块上接口链路异常，导
致HA心跳和同步报文传输异常，系统支持配置接口扩展模块上的接口作为HA控制连接接口。
指定HA控制连接接口，在全局配置模式下，使用以下命令：
ha link interface interface-name
l
interface-name – 指定接口名称。目前仅支持将物理接口、集聚接口或冗余接口配置为控制连接的接
口。用户最多可以指定1个集聚接口或冗余接口为HA控制连接接口，或者最多指定2个物理接口为HA控
制连接接口。
指定HA数据连接接口，在全局配置模式下，使用以下命令：

l
ha link data interface - 指定HA连接为数据连接。指定后，会话信息将通过HA连接接口同步完成。
l
interface-name – 指定接口名称。目前仅支持将物理接口、集聚接口或冗余接口配置为数据连接的接

指定HA数据连接接口工作模式
当HA数据连接接口配置为两个物理接口时，这两个物理接口可以工作在主备模式（Active-passive
Mode）或者负载模式（Load-balance Mode）。指定HA数据连接接口工作模式，在全局配置模式下，使
用以下命令：
ha data-link mode {active-passive | load-balance}
l
active-passive – 指定HA数据连接接口工作模式为主备模式。指定后，先配置的HA数据连接接口为主
接口，后配置的HA数据连接接口为备份接口。当主接口发生故障时，备份接口接替主接口工作；当主接
口恢复正常后，主接口重新运行所有工作。
l
load-balance – 指定HA数据连接接口工作模式为负载模式，该模式为系统默认模式。指定后，两个
HA数据连接接口互为备份，且同时运行各自的工作。当其中一个接口发生故障时，另一个接口运行其自
身的工作并且接管故障接口的工作；当故障接口恢复正常后，两个接口重新运行各自的工作。
在全局配置模式使用该命令no的形式恢复系统默认配置：
no ha data-link mode

指定HA连接接口的最大传输单元（MTU）
HA连接接口指定完毕后，用户可按需为HA连接接口配置最大传输单元（MTU）值。当报文大小超过配置的

ha link mtu value
l
value – 指定HA连接接口的最大传输单元的数值。默认为1500。

HA簇配置
为设备配置了HA组、HA组接口和HA连接接口后，用户需要将设备添加到HA簇中，才能够使设备的HA功能
生效。如果网络中存在多对HA设备，用户需要为它们配置不同的HA簇ID，否则可能出现MAC地址冲突现
象。配置HA簇，在全局配置模式下使用以下命令：
ha cluster cluster-id [[peer-mode node ID [symmetric-routing]]| node ID]
l
cluster-id – 指定HA簇ID。范围依据HA的虚MAC前缀而变化。
l
peer-mode node ID – 配置HA Peer模式，并标识该设备在HA簇中的角色。范围是0到1。默认情况
下，HA Node ID为0的设备上的组0为主动状态，HA Node ID为1的设备上的组0为禁用状态。
l
symmetric-routing - 若指定该参数，设备将工作在对称路由模式下。建议在只有对称流量的应用场
景下开启该功能，开启后系统会简化会话处理流程。该功能默认关闭，即设备默认使用非对称路由模
式。注意：对称路由配置和ha traffic enable互斥。
l
node ID - 为设备指定HA Node值，两台设备需指定不同的Node值。范围是0到1。云·界如不指定该
参数，设备将通过自动协商获取Node ID。其他Hillstone设备必须指定该参数。
在全局配置模式下使用no ha cluster关闭设备的HA功能。

启用/禁用HA Pinhole信息自动同步功能
一些应用程序（如FTP、SIP等）采用多通道数据传送，其控制通道和数据通道是分开的。数据通道的IP地址
和端口信息是提前由控制通道协商确定的。防火墙在开启ALG功能的情况下，系统会将数据通道的IP地址和
端口信息存储至Pinhole信息，数据包匹配Pinhole信息就可以通过防火墙，从而保证多通道应用程序正常
的通信。
默认情况下，工作在Active-Passive（A/P）模式的HA设备之间会自动同步Pinhole信息。同步Pinhole信
息会产生一定流量，在高负载情况下可能会对设备性能造成影响，用户可以根据设备负载情况启用或禁用HA
Pinhole信息自动同步功能，以确保设备的稳定性。
启用或禁用HA Pinhole信息自动同步功能，在全局配置模式下，使用以下命令：
l
启用：ha sync rdo pinhole enable
l
禁用：ha sync rdo pinhole disable

HA联动动态路由介绍
在网络系统中，HA联动动态路由是一项重要的功能，它能够保障网络在不同设备状态下的高效稳定运行。动
态路由协议如OSPF、OSPFv3（OSPF的IPv6版本）和BGP用于在网络中自动发现和计算最佳路由路径，而
HA联动动态路由则在此基础上，根据HA的主备状态对这些动态路由的cost值（成本值，用于衡量路由路径
的优劣，cost值越小表示路由路径优先级越高）进行调整。
HA联动动态路由功能可以实现：
l 主设备优先：正常时让主设备的低cost值路由被优先选用，承担主要流量，发挥其性能优势。
l 故障切换：主设备故障时，备设备能够调整自身路由cost值，吸引流量，快速接管工作，减少业务中断。
l 可靠稳定：自动调整路由路径，即便设备故障或网络异常，也能维持网络运行，降低管理难度和成本。
系统可以根据HA的主备状态动态调整OSPF、OSPFv3和BGP路由的cost值，具体调整如下：
主设备的路由发布
主设备将按照动态路由的cost值配置正常发布路由。即主设备会按照动态路由配置中的cost值，向网络中的
其他设备宣告自身所掌握的路由信息，以便其他设备能够根据这些信息选择合适的路径进行数据传输。

<!-- 来源页 992 -->
备设备的路由发布
备设备会将动态路由的cost值调整为指定数值并发布路由。目的是在正常情况下，使备设备的路由路径优先
级低于主设备，从而使得网络流量优先通过主设备进行处理，只有当主设备出现故障等异常情况时，备设备
才会承担起主要的数据传输任务。
如下图所示，两台防火墙均未开启路由联动功能，原OSPF路由的cost值为1，此时内外部网络之间的流量由
两台防火墙共同处理。其中M0D1设备是HA组0处于主动状态、HA组1处于禁用状态的设备；D0M1设备是
HA组1处于主动状态、HA组0处于禁用状态的设备。
如下图所示，两台防火墙均开启了路由联动功能并设置调整的cost值为65500，原OSPF路由的cost值为1。
此时主设备FW1仍会按照cost值为1发布路由，而备设备FW2将会按照cost值为65500发布路由。由于FW2
的路由cost值远大于FW1，其他设备在选择路由时会优先选择FW1，此时内外部网络之间的流量将由FW1处
理，从而实现了主设备承担流量的目的。

<!-- 来源页 993 -->
如下图所示，两台防火墙均开启了路由联动功能并设置调整的cost值为65500，原OSPF路由的cost值为1。
此时主设备FW1上行业务口故障，FW1切换为备设备，FW2切换为主设备。现在FW2会调整路由的cost值，

传输的连续性。

开始之前
在配置路由联动功能前，有一些重要的注意事项需要用户了解：
l
支持模式：路由联动功能仅支持HA Peer模式。HA Peer模式是一种HA Active-Active模式。处于HA
Peer模式下的两台设备都处于主动状态且同时运行各自的工作、相互监测对方的情况。
l
设备同步：路由联动功能不支持主备设备同步，需要分别在主备设备上单独配置，主备设备可以配置不
一样的路由cost值。如果有一端设备没有进行配置，则该设备就不会调整路由cost值。因此用户需要在
配置时对主备设备都进行相应的操作，以确保功能的正常实现。

l
cost值配置：用户在配置备设备的路由cost值时，需关注主设备原有的cost值。如果备设备配置的路由
cost值较小，小于主设备原有的cost值，则可能会影响业务运行。因为较小的cost值会使备设备的路由
路径优先级高于主设备，可能导致网络流量的不合理分配，进而影响业务的正常进行。

l
ospf-cost | ospfv3-cost | bgp-cost - 指定需要调整OSPF、OSPFv3或者BGP动态路由的cost值。
l
backup-cost-value - 指定动态路由所需调整的cost值。取值范围为1-65535，默认为65500。
关闭调整路由cost值功能，在全局配置模式下，使用以下命令：
ha adjust {ospf-cost | ospfv3-cost | bgp-cost} disable

通过UDP协议封装HA Traffic报文
在HA设备采用非物理直连，而是通过交换机连接的部署场景下，由于中间设备对流量进行哈希调度时存在不
均衡性，可能会限制防火墙的流量转发性能，无法达到满流量转发状态。为此系统支持通过UDP协议对HA
Traffic报文进行封装，确保HA Traffic报文在中间设备进行哈希计算时，其哈希值分布更为均匀，从而优化
流量在设备间的转发效率，提升HA架构下的流量吞吐性能。该功能默认为关闭状态。
注意:
l
通过UDP协议封装后，HA Traffic报文会多出40字节（IPv6报文会多出60字节），导致报文
更容易被分片/丢弃。
l
HA链路IP配置为IPv4地址时，HA Traffic报文可能会被中间设备分片；HA链路IP配置为IPv6
地址时，由于中间设备不会对IPv6报文进行分片，所以会丢弃超出MTU值的报文，从而出现丢
包等情况。
为保障业务正常运行，在通过UDP协议封装HA Traffic报文后，用户可以采取以下措施：
l
调整中间设备的接口MTU值，确保封装后的HA Traffic报文长度不超过接口MTU值。例
如：中间设备当前的接口MTU值为1500，则在IPv4场景下需调整为1540=1500+40，
在IPv6场景下需调整为1560=1500+60。
l
在中间设备上开启巨帧报文转发功能。为避免由于中间设备性能不足导致流量转发效率
低，建议用户调整中间设备的接口MTU值，不要分片过多。
开启通过UDP协议封装HA Traffic报文，在全局配置模式下，使用以下命令：
ha traffic transmit udp
在全局配置模式下，使用no ha traffic transmit udp命令，关闭通过UDP协议封装HA Traffic报文。

HSVRP
系统提供HSVRP（Hillstone Virtual Redundant Protocol）功能。在Peer Active-Active（A/A）场景
下，当一台设备发生故障时，它使得原本流向故障设备业务接口的流量可以流向另一正常运行的设备，由另
一台设备接管，从而保障网络通信的不间断。
HSVRP功能引入了HSVRP组的概念，一个HSVRP组由两台HA设备上对应的一组业务接口组成。HSVRP组
的虚拟IP地址将先在首选接口上生效，即访问虚拟IP地址的流量，将先流向HSVRP组的首选接口；当首选接
口所属设备故障时，从而流量被引流至另一台设备上的备选接口，使得网络中的通信不受故障设备的影响。

HSVRP典型场景
HSVRP通常部署在HA Peer模式中。以下图组网场景为例，M0D1设备是HA组0处于主动状态、HA组1处于
禁用状态的设备；D0M1设备是HA组1处于主动状态、HA组0处于禁用状态的设备。eth0/1和eth0/1:1分
别作为HA组0和组1的业务转发口。此时通过配置HSVRP功能，使得HSVRP组的虚拟IP先在eth0/1生效，
若M0D1发生故障，虚拟IP转移到eth0/1:1上生效，将访问eth0/1的流量引流至eth0/1:1，从而保障HA
组0的业务不受影响。
在如下场景中，配置HSVRP组，且HSVRP组的虚拟IP地址为20.1.1.1。该虚拟IP地址引用到业务接口
eth0/1和eth0/1:1，并首先在eth0/1上生效。配置内网PC的网关指向虚拟IP地址（20.1.1.1）。那么，
当PC向Server发起访问时：
l 若M0D1和D0M1均正常运行，虚拟IP地址（20.1.1.1）在eth0/1上生效，访问流量由M0D1转发。
l 若M0D1发生故障，虚拟IP地址（20.1.1.1）将转移到eth0/1:1上生效，D0M1接管M0D1的工作，转发访问流
量。

HSVRP基础概念
HSVRP组
HSVRP组是一个虚拟对象，具有一个虚拟IP地址和一个虚拟MAC地址。在HA场景下，配置好HSVRP组后，
需由HA设备上对应的一组业务接口对其进行引用，HSVRP组才可生效。
接口引用HSVRP组
接口引用HSVRP组后，HSVRP组的虚拟IP地址将生效。同一接口最多只能引用2个HSVRP组。当引用2个
HSVRP组时，接口需作为其中一个HSVRP组的primary接口和另一个HSVRP组的secondary接口。
若接口作为HSVRP组的primary接口，HSVRP组的虚拟IP地址将首先在该接口上生效；若接口作为HSVRP
组的secondary接口，只有当HSVRP组的primary接口所属设备发生故障时，HSVRP组才在此secondary
接口上生效。
举例来讲，在Peer A/A场景下，M0D1设备使用eth0/1作为业务转发口，D0M1设备使用eth0/1:1接口作
为业务转发口。存在两个配置好虚拟IP的HSVRP组：hsvrp 1和hsvrp 2。
l 示例1：接口eth0/1和eth0/1:1只引用hsvrp 1。eth0/1作为hsvrp 1的primary接口，eth0/1:1作为hsvrp 1的
secondary接口。
那么，hsvrp 1的虚拟IP将在eth0/1上生效，由eth0/1接收访问虚拟IP的流量，而eth0/1:1将处于备份状态。只
有当M0D1发生故障时，hsvrp 1的虚拟IP才在eth0/1:1上生效，eth0/1:1开始接管eth0/1的业务流量。
l 示例2：接口eth0/1和eth0/1:1同时引用hsvrp 1和hsvrp 2。eth0/1作为hsvrp 1的primary接口、hsvrp 2的
secondary接口；eth0/1:1作为hsvrp 1的secondary接口、hsvrp 2的primary接口。
那么，hsvrp 1的虚拟IP将在eth0/1上生效，hsvrp 2的虚拟IP将在eth0/1:1上生效。也就是说，同一时间，
eth0/1接收访问hsvrp 1虚拟IP的流量，eth0/1:1接收访问hsvrp 2虚拟IP的流量，且eth0/1和eth0/1:1互为备
份。
l 若M0D1发生故障，eth0/1:1将接管eth0/1的业务流量，访问hsvrp 1和hsvrp 2的虚拟IP的流量，将均到
达eth0/1:1。

<!-- 来源页 1015 -->
l 若D0M1发生故障，eth0/1将接管eth0/1:1的业务流量，访问hsvrp 1和hsvrp 2的虚拟IP的流量，将均到
达eth0/1。
HSVRP组的状态
系统根据HSVRP组的状态，判断HSVRP组的虚拟IP地址在哪个接口上生效，生效接口将处理访问虚拟IP的流
量。同一时间，只有一个接口生效。当生效接口所属设备发生故障时，另一台设备上的接口才接管故障设备
的流量，保障网络中业务的连续性。
HSVRP组的状态有两种：active和inactive。若接口上HSVRP组的状态为active，表示当前该接口为
HSVRP组的生效接口，将处理访问虚拟IP地址的流量；若接口上HSVRP组的状态为inactive，表示该接口
不是HSVRP组的生效接口，将由另一接口处理访问虚拟IP地址的流量。同一时间，HSVRP组在一个接口上的
状态为active，那么，在另一个接口上的状态一定为inactive。
HSVRP组在接口上的状态，由接口是否为HSVRP组的primary接口、接口所属设备是否正常运行决定：
l 若primary接口所属设备处于正常运行状态，则HSVRP组在primary接口上的状态为active。否则，HSVRP组将
在其secondary接口上的状态为active。
l 若primary接口所属设备从故障状态恢复正常运行，HSVRP组在primary接口上的状态将从inactive变回active。
当HSVRP组在接口上的状态从inactive变为active时，该接口将使用HSVRP组的虚拟IP和虚拟MAC地址发
送免费ARP报文，将访问虚拟IP的流量引流至自身，然后进行转发。

创建HSVRP组
目前，系统最多支持创建255个HSVRP组。在全局配置模式下，使用以下命令创建HSVRP组：
hsvrp id id
l
id - 指定该HSVRP组的ID号，取值范围是1到255。HSVRP组的ID号，用来区分不同的HSVRP组，如
hsvrp id 1和hsvrp id 2。执行该命令后，将进入HSVRP组的配置模式。
在全局配置模式下，使用no hsvrp id id命令删除指定的HSVRP组。

配置HSVRP组虚拟IP地址
HSVRP组的IP地址与其将要绑定的接口IP地址可以属于同一网段，也可以属于不同网段。在HSVRP组配置模
式下，使用以下命令：
ip address {A.B.C.D | A.B.C.D/M}
l
A.B.C.D | A.B.C.D/M - 指定HSVRP组的IPv4地址。
l
若指定为A.B.C.D格式，表示子网掩码为255.255.255.255，此时指定的IPv4地址需与将要绑定
的接口IPv4地址属于同一网段。
l
若指定为A.B.C.D/M格式，此时，配置的HSVRP组IPv4地址需与将要绑定的接口IP地址属于不同
网段。
在HSVRP组配置模式下，使用no ip address [A.B.C.D | A.B.C.D/M]命令取消配置。
ipv6 address {X:X:X:X::X | X:X:X:X::X/M}
l
X:X:X:X::X | X:X:X:X::X/M - 指定HSVRP组的IPv6地址。
l
若指定为X:X:X:X::X或X:X:X:X::X/128格式，指定的IPv6地址需与将要绑定的接口IPv6地址属
于同一网段。
l
若指定为X:X:X:X::X/M（M不等于128）格式，此时，配置的HSVRP组IPv6地址需与将要绑定的
接口IPv6地址属于不同网段。
在HSVRP组配置模式下，使用no ipv6 address 命令取消配置。
提示: 为HSVRP组配置IP地址后，系统将根据HSVRP组的ID号，自动生成该HSVRP组对应的虚拟
MAC地址。生成的虚拟MAC地址的格式为：00-00-5E-00-01-id，其中“id”表示HSVRP组ID号
的十六进制，如hsvrp 17对应的虚拟MAC地址为：00-00-5E-00-01-11。
查看HSVRP组
查看HSVRP组的信息，在任何配置模式下，使用以下命令：
show hsvrp [id id]
l
id id - 查看指定ID号的HSVRP组信息，包括虚拟IP地址、虚拟MAC地址、primary和secondary接口

接口引用HSVRP组
在Peer模式下，为实现M0D1设备或D0M1设备发生故障时，故障设备的业务接口地址仍然可达，相关接口
均需引用HSVRP组。接口引用配置好的HSVRP组后，将HSVRP组的IP地址向外提供，那么，当存在访问
HSVRP组IP地址的流量时，流量最终会到达引用该HSVRP组的接口，然后由该接口所属设备进行处理。
系统支持以下类型的接口引用HSVRP组：物理接口、集聚接口、冗余接口、VSwitch接口和上述接口的子接
口。
配置接口引用HSVRP组，先在全局配置模式下，使用以下命令进入该接口的接口配置模式：
interface interface-name
l
interface-name - 指定接口名称，并进入该接口的配置模式。
在接口配置模式下，使用以下命令为接口引用配置好的HSVRP组：
hsvrp id id {primary | secondary}
l
id - 指定HSVRP的ID号。
l
primary - 指定该接口为HSVRP组的primary接口。Peer A/A模式下的两台设备均正常工作的情况
下，引用的HSVRP组在该接口上为active状态。
l
secondary - 指定该接口为HSVRP组的secondary接口。Peer A/A模式下的两台设备均正常工作的情
况下，引用的HSVRP组在该接口上为inactive状态。
在接口配置模式下，使用no hsvrp id id {primary | secondary}命令取消引用的HSVRP组。
注意:
l
接口的IP地址与引用的HSVRP组的虚拟IP地址不可以相同。
l
接口引用HSVRP组后，接口本身的IP地址仍然可以正常使用。但是，建议仅作为管理IP使用，
不建议作为其他业务转发口使用。
l
当同一接口分别作为primary接口和secondary接口，与两个HSVRP组绑定时，两个HSVRP
组的IP地址必须属于同一网段。
l
若需引用HSVRP组，接口的IP地址必须是手工配置的静态IP地址，不可以使用动态获取功能。
l
接口若需引用HSVRP组，不可以配置管理IP或启用Local属性，接口的二级IP地址也不可以与
引用的HSVRP组的IP地址相同。

本示例介绍如何在HA Peer场景下，配置HSVRP功能，实现两台HA设备的接口作为不同的网关，一侧异常

l 当M0D1和D0M1正常运行时，PC1访问Server的流量由M0D1转发，PC2访问Server的流量由D0M1转发。
l 当D0M1发生故障时，在不需要更改PC2和Server网关配置的情况下，PC1和PC2访问Server的流量，均由M0D1
转发，业务流量不中断。

VRRP
虚拟路由冗余协议VRRP（Virtual Router Redundancy Protocol）是一种网络层协议，可以作为高可靠
性解决方案，在网络的默认网关出现故障时，及时将业务流量切换到备份网关设备，实现网关的冗余备份，
保证网络通信的可靠性。
VRRP包括VRRPv2和VRRPv3两个版本，VRRPv2版本只支持IPv4 VRRP，VRRPv3版本支持IPv4 VRRP和
IPv6 VRRP（VRRP6）。VRRP6（Virtual Router Redundancy Protocol for IPv6，IPv6虚拟路由器冗
余协议）主要用于提供IPv6网络中的默认网关冗余，确保网络的高可用性。
当前系统版本仅支持VRRPv3。
注意:
l
VRRP功能和HA功能互斥，不能同时配置。
l
VRRP功能采用标准的VRRP协议，可以与支持标准VRRP协议的其它厂商设备实现相互协商。
l
VRRP功能不支持配置同步以及会话同步，仅提供网关故障时的冗余备份。
l
仅支持通过CLI配置，参考《StoneOS命令行用户手册》。
VRRP典型场景
VRRP的典型应用场景主要包含主备备份及负载分担两种。
主备备份
主备备份应用场景包含一个虚拟路由器，该虚拟路由器由一台主设备和一台或者多台备份设备组成。主设备
承担报文转发任务，当主设备发生故障时，会从其它备份设备中根据优先级选举新的主设备，接替原来主设
备的工作，继续转发报文。

<!-- 来源页 1025 -->
如上图所示，Device A、Device B和Device C组成一个虚拟路由器，Device A为主设备并承担报文转发
任务，Device B和Device C为备份设备且不进行报文转发。Device A定期发送VRRP通告报文通知Device
B和Device C自己工作正常。如果Device A发生故障，Device B和Device C会根据优先级选出新的主设
备，作为网关继续转发报文。
负载分担
负载分担应用场景包含两个或者两个以上的虚拟路由器，多个虚拟路由器同时承担网关的功能。用户可以为
不同的主机指定不同的虚拟路由器作为网关，实现负载分担。
在负载分担应用场景中，同一台设备可以加入多个VRRP备份组，通过设置不同的优先级，该设备在一个备
份组中可以作为主设备，在其它备份组中作为备份设备。

如上图所示，Device A和Device B组成两个虚拟路由器Virtual Router 1和Virtual Router 2。Virtual
Router 1中，Device A为主设备，Device B为备份设备；Virtual Router 2中，Device A为备份设备，
Device B为主设备。局域网内的一部分主机将Virtual Router 1作为网关，另一部分主机将Virtual
Router 2作为网关，Virtual Router 1和Virtual Router 2均处理流量，实现业务流量的负载分担。如果其
中一个虚拟路由器出现故障，所有流量将会通过正常运行的虚拟路由器进行处理，同时又实现了相互备份。
VRRP基础概念
VRRP将局域网内的多台物理网关设备组成一台虚拟路由器，并将虚拟路由器的IP地址设置为默认网关地址，
局域网内的主机都通过虚拟路由器与外部网络进行通信。虚拟路由器的报文转发由组成虚拟路由器的物理网
关中的某一台设备承担，当该物理网关设备发生故障时，VRRP协议通过VRRP选举，从其它物理网关中选择
一台设备接管故障设备的工作，继续转发报文。

VRRP相关概念如下：
l 虚拟路由器（Virtual Router）：又称为VRRP备份组，由一台主设备和一台或者多台备份设备组成，对外相当于
一台虚拟路由器。如上图，Device A和Device B组成了一个虚拟路由器。
l 虚拟路由器ID（Virtual Router ID，VRID）：用于标识虚拟路由器。如上图，Device A和Device B组成的虚拟
路由器的VRID为1。
l 主设备（Virtual Router Master ）：虚拟路由器中承担报文转发工作的设备。如上图，Device A为主设备。
l 备份设备（Virtual Router Backup）：主设备出现故障时，能够替代主设备进行报文转发的设备。如上图，
Device B为备份设备。
l 虚拟IP地址（Virtual IP Address）：虚拟路由器的IP地址。VRRP6备份组的虚拟IP地址包含IPv6链路本地地址和
全球单播地址。如上图，Device A和Device B组成的虚拟路由器的虚拟IP地址为192.168.1.1/24。
l IP地址拥有者（IP Address Owner）：如果一个VRRP备份组内的某台设备的接口IP地址和虚拟路由器的IP地址
相同，该设备被称为IP地址拥有者。当设备为IP地址拥有者时，其优先级始终保持255，只要该设备处于工作正
常状态，会一直保持为主设备。
l 虚拟MAC地址（Virtual MAC Address）：一个虚拟路由器拥有一个虚拟MAC地址。IPv4 VRRP虚拟路由器的
MAC地址格式为“00-00-5E-00-01-VRID”；VRRP6虚拟路由器的MAC地址格式为“00-00-5E-00-02-
VRID”。一般情况下，当虚拟路由器回应ARP请求时，使用虚拟MAC地址，而不是接口的真实MAC地址。如上

<!-- 来源页 1028 -->
图，Device A和Device B组成的虚拟路由器的VRID为1，因此，该IPv4 VRRP备份组的MAC地址为00-00-5E-
00-01-01。
l 优先级（Priority）：用于VRRP选举，VRRP备份组中优先级高的会被选举为主设备。VRRP优先级的取值范围为
1到254，数值越大表明优先级越高，255为系统保留给IP地址拥有者的优先级。
l 抢占模式（Preempt Mode）：如果将VRRP备份组中的设备配置为抢占模式，一旦该设备发现自己的优先级高
于主设备，就会将自己升级为主设备，而原先的主设备将变为备份设备；如果将设备设置为非抢占模式，即使该
设备的优先级高于主设备，它也只能在主设备故障时代替主设备工作。
配置VRRP功能
使用VRRP功能，需要按照以下步骤进行配置：
1. 配置VRRP备份组。VRRP备份组的配置包括配置VRRP备份组的虚拟IP地址、指定备份组中设备的抢占模式和优先
级、配置VRRP通告报文的发送时间间隔。
2. 将VRRP备份组绑定到接口。
提示: 本章节中提及的VRRP为IPv4 VRRP和IPv6 VRRP（VRRP6）的统称。例如，VRRP备份组包
含IPv4 VRRP备份组和IPv6 VRRP备份组。
前提条件
l 将VRRP备份组绑定到接口前，请先为接口配置IP地址（IPv4地址为主IP地址，IPv6地址为全球单播地址）。
l 为接口配置IPv6全球单播地址前，需要先开启接口的IPv6功能。具体操作，请参阅IPv6地址配置。

配置抢占模式
如果将VRRP备份组中的设备配置为抢占模式，一旦该设备发现自己的优先级高于主设备，就会将自己升级
为主设备，而原先的主设备将变为备份设备；如果将设备设置为非抢占模式，即使该设备的优先级高于主设
备，它也只能在主设备故障时代替主设备工作。在配置抢占模式时，用户还可以设置延迟时间，使备份设备
在延迟时间过后升级为主设备。配置抢占模式，在VRRP备份组配置模式下，使用以下命令：
preempt [delay delay-time]
l
delay delay-time - 指定延迟时间，取值范围为0到3600秒，默认值为0。
在VRRP备份组配置模式下，使用no preempt命令恢复为非抢占模式。
指定优先级
该命令指定的优先级用于VRRP选举。在一个VRRP备份组中，优先级高的设备会被选举为主设备，数值越大
表明优先级越高。为设备指定优先级，在VRRP备份组配置模式下，使用以下命令：
priority number
l
number - 指定优先级。范围是1到254之间的整数。默认值是100。
在VRRP备份组配置模式下，使用no priority命令将优先级恢复为默认值。
配置VRRP通告报文的发送时间间隔
VRRP备份组中的主设备会定期发送VRRP通告报文，通知组内的备份设备自己的工作状态。如果备份设备在
Master Down时间间隔内没有收到VRRP通告报文，则会进行VRRP选举，选择新的主设备。
Master Down时间间隔的计算方法为：3 * VRRP通告报文发送间隔+ Skew_Time。其中，Skew_Time的
计算方法为：（（256 - 设备在备份组中的优先级）* VRRP通告报文发送时间间隔）/256。
配置VRRP通告报文的发送时间间隔，在VRRP备份组配置模式下，使用以下命令：
advertise-interval interval-value
l
interval-value - 指定VRRP通告报文的发送时间间隔。取值范围为1到40秒，默认值为1秒。

孪生模式
目前在很多行业中，数据中心承载了重要的数据信息以及办公业务。为了提高可靠性，出于灾备的目的，企
业一般都会建设两个或多个数据中心，并采用二层扩展方式（大二层技术或者物理光纤直连）将两个数据中
心进行二层互联，两端数据中心同时独立运行、对外提供业务服务并且互为备份，构成双活数据中心。
Hillstone防火墙以路由模式部署方式部署在数据中心中，用来检查跨越不同区域的流量以及实现不同区域
的策略隔离。由于两个数据中心之间由大二层技术互通，跨越数据中心且跨越不同区域的三层流量会出现非
对称路由的情况（即进出流量可能经过不同路径），使得防火墙的策略隔离功能不能正常使用。
为了解决上述这一问题，系统提供孪生模式（Twin-mode HA）功能，从而提供更优化的流量数据转发功
能，使两端数据中心防火墙实现策略统一部署，保障双活数据中心的业务持续性、高效性。

l
配置孪生模式，需要提前安装Twin-mode许可证。
l
目前仅孪生模式A/P模式的连接接口和对端支持IPv6地址配置。
l

指定非对称路由流量转发模式
针对非对称路由流量，孪生模式功能提供两种转发模式，分别是隧道模式和二层隧道模式。
l
隧道模式：一端数据中心将数据包封装后通过数据连接（Data Link）发送到对端数据中心，对端数据中
心对解封装后的流量进行转发。默认情况下，使用该模式转发流量。
l
二层隧道模式：该模式是将数据包目的MAC地址修改为对端数据中心对应接口的虚拟MAC（VMAC）地
址，流量通过二层隧道转发到对端数据中心。使用该模式，用户需要在设备的所有业务接口开启二层隧
道转发模式。
开启二层隧道转发模式，在接口配置模式下，使用以下命令：
twin-mode-l2-tunnel-enable
使用no twin-mode-l2-tunnel-enable命令恢复默认转发模式：隧道模式。
配置孪生模式首包转发Bounce Back功能
在孪生模式A/A模式部署场景中，用户可以配置首包转发Bounce Back功能。配置该功能后，本端设备先后
将新建session同步信息和触发新建的首包发送给孪生模式对端设备，当对端设备在完成session创建之
后，会将收到的首包返回至本端设备，之后由本端设备进行下一步转发处理，从而解决非对称路由环境下，
由于对端设备session同步不及时导致反向报文找不到session丢包的问题。
配置孪生模式首包转发Bounce Back功能，在Twin-mode配置模式下，使用以下命令：
first-packet-bounce-back [max-size num]
l
max-size num – 指定首包最大尺寸。取值范围为64到1024字节。若不指定该参数，则默认首包最大
尺寸为128字节。
在Twin-mode配置模式下，使用no first-packet-bounce-back取消配置孪生模式首包转发Bounce
Back功能。

配置孪生模式网关功能
在网关模式部署下，由于二层扩展设备会过滤两端数据中心网关相同的IP地址和MAC地址，导致非对称流量
不通的问题，为了避免此问题，用户需要配置孪生模式网关功能，指定网关接口发送ARP请求报文的IP地
址，系统将以此IP地址作为源IP、以Twin-mode 虚拟MAC（VMAC）地址作为源MAC地址发送ARP请求报
文；以Twin-mode虚拟MAC（VMAC）地址作为源地址转发数据流量，从而解决网关模式下的非对称流量
问题。
开启接口的孪生模式网关功能并指定发送报文的IP地址，在接口配置模式下，使用以下命令：
twin-mode-gateway sender-ip ip-address
l
ip-address – 指定网关接口的发送ARP请求报文的IP地址。该地址必须与网关接口的IP地址在同一网
段。
使用no twin-mode-gateway sender-ip ip-address命令关闭该功能并且删除指定的发送IP地址。
注意: 两端数据中心网关接口必须配置不同的IP地址用来发送ARP请求报文。
指定孪生模式会话状态切换方式
在孪生模式A/A部署方式下，系统支持两种会话状态切换方式，分别是单向切换和双向切换。
l
单向切换：当一端数据中心访问外网服务器的链路故障时，系统将非活跃（inactive）会话状态快速切
换到活跃（active）状态，能够保证流量不间断。

跨型号组HA/孪生模式
在企业运营及数据中心管理过程中，时常面临设备更新换代的需求。当新设备替换旧设备时，传统做法是新
设备上线后，旧设备随即下线处理。
为更好地保障设备的可靠性，山石网科推出跨型号组HA和跨型号组孪生模式，新设备作为主设备正常进行业
务转发，旧设备作为备份设备。运行一段时间，如果新设备运行状态正常，则下线旧设备，保障新设备的平
滑替换。满足单出口设备替换以及多出口设备替换等多个场景。
跨型号涵盖同系列设备的不同型号以及不同系列设备之间的HA/孪生模式组网。
组HA/孪生模式的机制
状态协商和数据同步是设备组HA和孪生模式的两大核心机制。
l 状态协商：主设备和备份设备通过心跳机制实时交换状态信息，确保双方状态一致。
l 数据同步：主设备通过冷同步和热同步的方式将数据同步至备份设备，确保在主设备发生故障时，备份设备能够
完整接管业务，避免数据丢失。数据同步，包括同步配置信息、文件信息以及RDO信息。
在跨型号组建HA时，受限于设备型号差异，设备间仅执行状态协商流程，而不会开展数据同步操作。跨型
号组孪生模式，只要有一台设备型号不同，两个数据中心之间只进行状态协商，不进行数据同步。当主设备
出现故障，触发主备切换流程，备份设备将通过新建会话的方式，重新构建业务连接，进而实现业务的顺利
接管。

单出口设备替换
在企业网络环境中，经常面临设备更新换代的需求。假设企业当前网络中正在使用一台型号为A的山石防火
墙设备，因为业务拓展、性能升级等需求，新购置了一台型号为B 的山石网科防火墙设备。需要将新设备B
设备替换原有设备A 上线。以满足不断变化的网络安全和性能需求。
由于设备A和设备B是不同型号的山石防火墙设备，为了在设备替换过程中最大程度减少对业务的影响，可借
助跨型号组HA功能来实现平滑过渡。

<!-- 来源页 1052 -->
以下为具体的实现流程：
1. HA协商建立主备关系：首先将新设备B接入企业网络，并与正在运行的设备A进行HA协商操作。设备A和设备B
固件版本相同且均开启多VR特性。在协商过程中，系统会根据设备的相关参数和配置，确定设备A为主设备，设
备B为备份设备。
2. 主备切换实现设备替换：在设备A上执行主备切换操作。通过这一操作，设备B从备份设备转变为新的主设备，而
原主设备则变为备份设备。这样，原本由设备A处理的网络流量和业务任务，就会全部转移到设备B上进行处理。
3. 观察设备B运行情况并决策：设备B成为新主设备后，让其持续运行一段时间的业务。在这段时间内，网络管理员
需要密切关注设备B的运行状态，包括设备的性能指标（如CPU利用率、内存利用率等）、网络流量情况等。
l 如果在观察期内设备B发生故障，需要立即将网络流量切回设备A。让设备A重新承担起主设备的职责，继
续处理业务。
l 如果一段时间内设备B运行正常，说明设备B能够稳定地满足企业网络的需求。此时，可以将设备A下线处
理，完成整个设备的替换过程。

3. 观察新主数据中心运行情况并决策：数据中心B作为新的主数据中心运行一段时间，监控其性能指标（如CPU利

l 如果一段时间内新主数据中心运行正常，各项指标均满足企业网络需求，则将备数据中心的两台SG-
6000-A5500设备也按相同流程替换为SG-6000-X20812 设备，进一步提升整个数据中心出口设备的性能

Bypass功能
为了避免单点故障而引起网络中断，部分型号设备支持使用Bypass模块。通过使用Bypass模块，当设备在
特定状态下（如系统重启、工作异常、设备断电）不能正常转发网络流量时，系统将启动Bypass模式，让
无法通过设备系统实现互联的两个网络，直接在物理上连通。
Bypass模块的连接方式

l 对于设备接口本身支持Bypass功能情况，用户只需将互联的两个LAN（如LAN1，LAN2）连接到一个Bypass接

设备在特定状态下（如系统重启、工作异常、设备断电）不能正常转发网络流量时，系统将进入Bypass状
态。Bypass状态下，互为Bypass的接口对在物理上直接连通，两接口之间相当于连接了一根网线，流量将
直接通过。设备正常运行状态下，Bypass接口对并不会连通，而是按照接口配置的功能正常转发流量。
Bypass接口对举例：设备E0/0与E0/1默认为Bypass接口对0，即pair 0，在前面板中设有标识；IOM-
8MM-B-260模块上有16个光接口，支持4对Bypass，即与设备上其他模块相连的Bypass接口对有4对，分
别为pair0（接口0'和接口1'）、pair1（接口2'和接口3'）、pair2（接口4'和接口5'）和pair3（接口6'和
接口7'）。

监控接口运行状态
设备在特定状态下（如系统重启、工作异常、设备断电）不能正常转发网络流量时，系统将进入Bypass状
态。IOM-8MM-B-260模块上有4对Bypass接口对，可以分别监控4组设备其他扩展模块上的接口。例如，
IOM-8MM-B-260模块上的pair0与设备上另一模块的接口“interface1”和“interface2”相连，当系统
监控到该模块接口“interface1”或“interface2”不能正常工作时，pair0对应的接口0和接口1将直接连
通，即该Bypass接口对处于Bypass模式。
1.
在全局配置模式下，使用以下命令进入相应Bypass接口对的配置模式：
bypass slot slot-id pair pair-id
